CloudFront managed prefix list: AWS maintains com.amazonaws.global.cloudfront.origin-facing containing all CF edge IP ranges. Using this as the ALB inbound source ensures only CloudFront — not arbitrary internet IPs — can directly reach the ALB. Prevents WAF bypass.
User Journey
Request Flow — Browser to Database
Every hop annotated. 8 steps from browser to database and back. Every AWS service label explained in context.
1
Browser — DNS lookup
Browser resolves partners.yourapp.com via OS DNS → ISP → Route 53 authoritative NS.
Route 53 returns CloudFront distribution domain (d1xxxxx.cloudfront.net) as A ALIAS record. Browser connects to nearest CF PoP via TCP + TLS 1.3.
R53 · A ALIAS → CF dist.TTL 60s
2
CloudFront Edge PoP — cache + WAF
TLS terminates at the nearest CF edge (not in Ohio — saves ~150ms for distant users). CF checks cache. API calls have TTL=0, so all API requests are cache MISSes and forward to ALB. Static assets (JS, CSS) are cache HITs — served instantly.
WAF WebACL evaluated on every request: OWASP Managed Rules + rate limit (2,000 req/IP/5min). WAF BLOCK → CF returns 403. WAF ALLOW → CF forwards to ALB.
TLS 1.3 terminationWAF: OWASP + rate limitCache HIT: static assets served here, EC2 not involved
3
ALB — Listener Rule Evaluation
ALB has one HTTPS listener (port 443). Rules evaluated in priority order. Receives request with Host: partners.yourapp.com header (preserved by CF). Matches Rule → forwards to target group tg-fe-3001 on port 3001. ALB SG only accepts traffic from CloudFront managed prefix list.
Host: partners.yourapp.com → TG:3001Public subnet · ACM cert
4
FE EC2 · Caddy → partner-portal container
Traffic arrives at FE EC2 on port 3001. Caddy (Docker container) proxies to the partner-portal Next.js container. Next.js renders the page (SSR or static). ALB→EC2 traffic is HTTP within the private VPC — TLS not needed internally.
Private subnet · 10.0.10.xt3.medium · DockerPort 3001
5
API call → api-user / api-order
Browser JS calls api.yourapp.com/api/v1/orders. CF#4 → ALB → path rule /api/v1/orders* → api-order EC2 on port 8009. JWT verified by Express middleware before controller runs.
api-order · port 8009JWT verified by middleware
6
RDS PostgreSQL query
api-order connects to RDS on port 5432 using credentials from Secrets Manager. RDS is in isolated subnet — no internet route. Only EC2 SG allowed on port 5432. TLS-encrypted. Parameterized queries only — no string concatenation. Connection pooled via pg.Pool (max 20 connections per service).
Isolated subnet · 10.0.20.xPort 5432 · TLSParameterized SQL · no ORM
7
Redis — sessions + rate limiting
JWT middleware checks Redis first: session validity + rate limit counters. Redis key session:{userId} — 7-day TTL. Rate key rate:global:{ip} — 60s sliding window. Sub-millisecond response. Only hits RDS if Redis confirms session is valid.
Port 6379 · TLS + AUTHIsolated subnet
8
Response returned to browser
JSON response: api-order → ALB → CF edge → browser. CF caches static assets (JS/CSS/images) at edge for subsequent requests. API responses: Cache-Control: no-cache (TTL=0). Full round-trip: ~80–200ms depending on user location.
CF caches /_next/static/*API: TTL 0 (no cache)
Why CloudFront in front of ALB?
TLS terminates at nearest edge PoP. A user in LA gets TLS handshake at LA, not Ohio — saves ~150ms. Static assets cached globally. WAF blocks bad traffic before it reaches EC2.
How does the browser know the right EC2?
It doesn't — and doesn't need to. Browser only talks to CloudFront. CF talks to ALB. ALB routes by Host header. EC2 private IPs (10.0.10.x) are never exposed.
File uploads — S3 bypass flow
User picks file → browser calls POST /api/v1/orders/upload-url → api-order calls S3 SDK: getSignedUrlPromise('putObject') (15 min TTL) → returns presigned URL to browser → browser PUT directly to S3. EC2 never sees file bytes. api-order stores the S3 key in RDS.
CI/CD
Deployment Pipeline
Every git push to main triggers this pipeline automatically. Zero manual steps. No SSH keys stored anywhere. Every command shown.
1
git push
Developer pushes to main. GHA webhook fires in <1s.
2
OIDC Auth
GHA requests short-lived token from AWS IAM via OIDC. No long-lived keys in GitHub.
3
Detect Changed
ci-affected.sh diffs HEAD~1..HEAD. Only changed apps rebuild.
4
Docker Build
Multi-stage build on GHA runner. Stage 1: pnpm install + build. Stage 2: copy dist/ only. EC2 never runs npm.
5
Push ECR
Image tagged :latest and :$SHA. SHA tag enables rollback.
6
Migrate DB
pnpm migrate:up runs before any deploy. Fail = no deploy.
7
SSM Command
GHA calls aws ssm send-command → EC2 runs docker pull + compose up. No SSH.
8
Health Check
GHA polls /healthz every 10s for 3 min. Fail = rollback to :previous-SHA.

Step-by-Step with Real Commands

2
OIDC Auth — GHA assumes AWS IAM Role
GHA runner requests OIDC JWT from token.actions.githubusercontent.com → calls sts:AssumeRoleWithWebIdentity. AWS IAM verifies: issuer = GitHub IDP, sub = repo:YourOrg/Polaris:ref:refs/heads/main. Returns 1-hour temp credentials. No AWS_ACCESS_KEY_ID ever stored in GitHub.
- uses: aws-actions/configure-aws-credentials@v4
  with:
    role-to-assume: arn:aws:iam::123456789:role/polaris-gha-deploy
    aws-region: us-east-2
    role-session-name: gha-deploy-${{ github.sha }}
4
Docker Multi-Stage Build
Stage 1 (builder): full Node.js 22 image, installs all deps, runs pnpm build, produces dist/. Stage 2 (runner): slim Node.js 22-alpine, copies only dist/ + prod node_modules. No build tools, no source, no dev deps.
# Stage 1: builder
FROM node:22-alpine AS builder
RUN corepack enable && pnpm install --frozen-lockfile
RUN pnpm -F @polaris/api-user build

# Stage 2: runner (slim)
FROM node:22-alpine AS runner
COPY --from=builder /app/apps/api-user/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY devops/scripts/entrypoint.sh ./entrypoint.sh
ENTRYPOINT ["./entrypoint.sh"]
5
ECR Push — :latest + :SHA tags
aws ecr get-login-password --region us-east-2 | \
  docker login --username AWS --password-stdin $ECR_REGISTRY

docker push $ECR_REGISTRY/polaris/api-user:latest
docker push $ECR_REGISTRY/polaris/api-user:${{ github.sha }}
7
SSM Run Command — Tell EC2 to Pull and Restart
GHA calls aws ssm send-command targeting EC2 instance ID (stored in SSM Parameter Store). No SSH keys. SSM Agent on EC2 receives command, executes as root. GHA polls for completion status.
aws ssm send-command \
  --instance-ids $(aws ssm get-parameter --name /polaris/prod/ec2/api-user-id --query Parameter.Value --output text) \
  --document-name AWS-RunShellScript \
  --parameters '{"commands": [
    "aws ecr get-login-password --region us-east-2 | docker login --username AWS --password-stdin $ECR_REGISTRY",
    "docker compose -f /opt/polaris/docker-compose.prod.yml pull api-user",
    "docker compose -f /opt/polaris/docker-compose.prod.yml up -d --no-deps api-user"
  ]}'

FE Only: S3 Sync + CloudFront Invalidation
aws s3 sync .next/static/ s3://polaris-prod-static/_next/static/ \

aws cloudfront create-invalidation \

entrypoint.sh — Fetch Secrets at Container Startup

#!/bin/sh
# Runs before node app starts. Requires AWS_REGION + EC2 instance profile.

SECRET_JSON=$(aws secretsmanager get-secret-value \
  --secret-id polaris/prod/db \
  --region $AWS_REGION \
  --query SecretString \
  --output text)

export DB_HOST=$(echo $SECRET_JSON | jq -r .host)
export DB_PORT=$(echo $SECRET_JSON | jq -r .port)
export DB_NAME=$(echo $SECRET_JSON | jq -r .name)
export DB_USER=$(echo $SECRET_JSON | jq -r .user)
export DB_PASS=$(echo $SECRET_JSON | jq -r .password)

# ... repeat for jwt, redis, stripe, etc.

# Replace shell process with Node app (PID 1 = correct SIGTERM handling)
exec node dist/server.js
Why exec? Using exec node replaces the shell process with Node.js as PID 1. SIGTERM from Docker stop reaches Node.js directly for graceful shutdown. Without exec, the shell is PID 1 and Node.js is a child that doesn't receive Docker stop signals.

CloudFormation Stacks
10 independent stacks · Cross-stack references via SSM Parameter Store (not CF Exports — avoids deletion locks) · Deploy or delete any stack independently.
Why SSM Parameter Store instead of CloudFormation Exports? CF Exports create hard dependencies — you cannot delete Stack A if Stack B imports its export. SSM Parameters have no such constraint. Each stack reads SSM at deploy time, not at template parse time. This allows deleting and rebuilding any single stack without breaking others.

Why EC2 Ports Are Not Public

EC2 instances have no public IP address. They live in private subnets (10.0.10.x, 10.0.11.x). The only inbound rule on fe-ec2-sg and be-ec2-sg is from alb-sg — meaning only the ALB can initiate connections to the EC2 containers. No port scanners, bots, or internet traffic can reach EC2 directly, even if they somehow knew the private IP.
