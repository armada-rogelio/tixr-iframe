All traffic stays inside the VPC. Only ALB and CloudFront face the internet. Isolated subnets have zero internet route. Regional services (S3, Secrets Manager, KMS) are not zonal — AWS replicates them across multiple AZs in us-east-2 and exposes one regional endpoint. They are shown below the VPC in a dedicated band, not inside any AZ. ECS-on-EC2 + warm-host cold standby — api-gateway + 2 backend services (api-user, api-order) run as bin-packed ECS tasks on one cluster host in 2a. A second warm cluster host in 2b runs 0 tasks at steady state — image cached, ready to receive tasks within ~40–60s on AZ-a failure.

User Journey
Request Flow — Browser to Database
Every hop annotated. 9 steps from browser to database and back. ECS-on-EC2 · ALB routes all API traffic to api-gateway task (host-header condition) · gateway proxies to backend services via ECS Service Connect (polaris.local Cloud Map DNS). FE deferred — only API path active.
1
Browser — DNS lookup
Browser resolves partners.yourapp.com via OS DNS → ISP → Route 53 authoritative NS.
Route 53 returns CloudFront distribution domain (d1xxxxx.cloudfront.net) as A ALIAS record. Browser connects to nearest CF PoP via TCP + TLS 1.3.
R53 · A ALIAS → CF dist.TTL 60s
2
CloudFront Edge PoP — cache + WAF
TLS terminates at the nearest CF edge (not in Ohio — saves ~150ms for distant users). CF checks cache. API calls have TTL=0, so all API requests are cache MISSes and forward to ALB. Static assets (JS, CSS) are cache HITs — served instantly.
WAF WebACL evaluated on every request: OWASP Managed Rules + rate limit (2,000 req/IP/5min). WAF BLOCK → CF returns 403. WAF ALLOW → CF forwards to ALB.

ALB has one HTTPS listener (port 443). Rules evaluated in priority order. Receives request with Host: api.yourapp.com header (preserved by CF). Priority 1: Host=api.yourapp.com matches → forwards to tg-api-gateway (target type ip) → api-gateway task ENI :3000. The gateway handles all per-path routing internally via gateway.config.json — ALB no longer does per-service path routing. ALB SG only accepts traffic from CloudFront managed prefix list. Why CloudFront in front of ALB?
TLS terminates at nearest edge PoP. A user in LA gets TLS handshake at LA, not Ohio — saves ~150ms. Static assets cached globally. WAF blocks bad traffic before it reaches EC2.
How does the browser know the right ECS task?
It doesn't — and doesn't need to. Browser only talks to CloudFront. CF talks to ALB. ALB routes by Host header + path to the right target group, which lists task ENI IPs registered by ECS. Task ENI IPs (10.0.10.x / 10.0.11.x) are never exposed externally.
File uploads — S3 bypass flow
User picks file → browser calls POST /api/v1/orders/upload-url → api-order task calls S3 SDK: getSignedUrlPromise('putObject') (15 min TTL) → returns presigned URL to browser → browser PUT directly to S3. Task never sees file bytes. api-order stores the S3 key in RDS.

EC2 Instances — What Runs Where
2 EC2 instances act as ECS cluster hosts. AZ-a host runs 3 bin-packed ECS tasks (api-gateway, api-user, api-order). AZ-b host runs warm with 0 tasks at steady state — image cached, ready to receive tasks within ~40–60s on AZ-a failure. Cluster ASG scales hosts when capacity provider utilization passes 75%. Frontend deferred for now.
EC2 cluster host #1 — us-east-2a (t3.medium)
Capacity: 2048 CPU units · 4096 MB · packed 1536/2048 CPU + 3072/4096 MB (~25% headroom)

ECS tasks running (each 512 CPU / 1024 MB):
api-gateway task Express proxy :3000 · Service Connect
api-user task Express :8008
api-order task Express :8009

Each task gets own ENI (awsvpc mode) — own private IP, own SG, registered as ALB ip-type target.

Host IAM (instance profile):
• ECR: GetAuthorizationToken, BatchGetImage, GetDownloadUrlForLayer
• SSM: ssmmessages:* (host shell access)
• CloudWatch: logs:PutLogEvents
• ECS: container instance registration

Task IAM (per-task role, scoped):
• Secrets Manager: GetSecretValue scoped to that service's secret prefix
• CloudWatch: per-task log group writes
EC2 cluster host #2 — us-east-2b (t3.medium) warm · 0 tasks
Steady state: 0 tasks. ECS agent registered with cluster, host healthy. Docker image cache populated after first deploy.

Why warm host + 0 tasks: Cheapest meaningful HA without active-active. Trade ~40–60s of 5xx during AZ-a failover for ~$50/mo savings vs running warm tasks. Image cache skips ECR pull on failover (~30–90s saved). Host already running skips EC2 boot (~30–60s saved).

Failover path (AZ-a down):
1. ALB drops 2a targets (~30s health check)
2. ECS scheduler detects host disconnect (~30s)
3. Scheduler places 2 tasks on host #2 (only option)
4. Docker pulls from local cache → docker run (~10s)
5. ALB health checks pass on new task IPs (~30s)
6. Traffic flows to 2b · ~75s end-to-end, ~45s of 5xx

Infrastructure as Code
CloudFormation Stacks
11 independent stacks · Cross-stack references via SSM Parameter Store (not CF Exports — avoids deletion locks) · Deploy or delete any stack independently.
Why SSM Parameter Store instead of CloudFormation Exports? CF Exports create hard dependencies — you cannot delete Stack A if Stack B imports its export. SSM Parameters have no such constraint. Each stack reads SSM at deploy time, not at template parse time. This allows deleting and rebuilding any single stack without breaking others.

Why EC2 Ports Are Not Public

ECS cluster hosts (EC2) and task ENIs have no public IPs. They live in private subnets (10.0.10.x, 10.0.11.x). The only inbound rule on api-gateway-task-sg is from alb-sg. The only inbound rules on api-user-task-sg / api-order-task-sg are from api-gateway-task-sg — meaning only api-gateway can reach backend tasks. The cluster host SG (ecs-host-sg) accepts no app inbound at all. No port scanners, bots, or internet traffic can reach tasks or hosts directly, even if they somehow knew the private IPs.
